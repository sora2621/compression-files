# 高速化ベンチマーク結果

計測日: 2026-07-15  
環境: Windows / 12 logical CPUs / 16 GB RAM / Sharp 0.34.5 / libvips 8.17.3

固定フィクスチャは `npm run benchmark:generate` で生成し、すべての計測で同じ入力を使用した。ログには固定ID、工程、経過時間、サイズ、品質指標だけを記録し、元ファイル名と絶対パスは含めていない。

## 概要

| 経路                       |            修正前 |           修正後 |              変化 |
| -------------------------- | ----------------: | ---------------: | ----------------: |
| 全固定ワークロード（fast） |          39.132秒 |         26.481秒 |         32.3%短縮 |
| 24MP JPEG WebPエンコード   |           8.512秒 |          5.456秒 |         35.9%短縮 |
| 10秒1080p 動画処理         |           6.427秒 |          5.272秒 |         18.0%短縮 |
| 動画目標容量処理           | 10.639秒（2パス） | 4.746秒（1パス） |         55.4%短縮 |
| メタデータのみ削除         |           0.313秒 |          0.170秒 | stream copyを維持 |

`balanced` は従来と同じ品質値、エンコーダー設定、2パス処理を維持し、全体33.187秒だった。修正前との差にはOSキャッシュや実行時負荷による変動を含むため、アルゴリズム上の改善値には含めていない。

品質値、CRF、最低品質、安全な入力ピクセル数制限は変更していない。fastはエンコーダーの速度努力量と、目標容量処理のパス数だけを変更する。厳密な容量精度を優先する場合は`balanced`または`maximum-compression`を使用する。

透過PNGのlossless候補は修正前後ともRGBA画素比較が不一致だったため、このケースを「無劣化検証成功」とは記録していない。高速化後だけ判定基準を緩める変更は行っていない。

## ボトルネックと対策

- 最大のボトルネックは、目標容量動画の2パスエンコード（合計10.639秒）と高努力量の画像エンコード（8.512秒）だった。
- 通常動画のリサイズ、ノイズ除去、シャープ化、色補正は1本のフィルターチェーンへ統合し、1回のFFmpegプロセスで処理する。
- メタデータ削除だけの場合は再エンコードせずstream copyを使う。
- 画像で既取得のSharp metadataを再利用し、同じヘッダー解析を繰り返さない。
- 同一ボリューム内の一時出力移動はcopy + unlinkではなくrenameを使う。
- ffprobe結果はfileId単位、FFmpegの形式・codec・filter・hardware情報はTTL付きの共有キャッシュで再利用する。
- AIワーカーは常駐プロセスとモデルキャッシュを使用する。CUDA時はFP16とinference modeを使い、タイルサイズをGPUメモリから選ぶ。

## ハードウェアと同時実行

FFmpegにはNVENCとQuick Syncが含まれていたが、固定動画での実行確認ではNVENCの初期化に失敗し、Quick Syncは利用できた。`fast`は実行確認に成功したQuick Syncだけを採用し、初期化・エンコード失敗時はCPUへフォールバックする。AI用Python/CUDAモデルはこの計測環境では利用できなかったため、AIモデル読み込み・推論時間は`unavailable`として記録した。

12 CPU / 16 GB / 1 GPUとして検出した現在の上限は、画像4件、CPU動画1件、GPU動画1件、AI 1件/GPU、音声12件。CPU・メモリ・GPU重みを使うFIFOキューで、小さな画像ジョブが先に待つ長時間動画を追い越さない。
