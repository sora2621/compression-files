# Compression Files

画像・動画・音声を実データから判定し、容量削減、画質補正、形式変換、メタデータ削除、任意のAI高画質化を行う Next.js アプリです。

対応形式を拡張子の固定リストで決めません。起動中のFFmpegに `-formats`、`-demuxers`、`-muxers`、`-decoders`、`-encoders`、`-filters`、`-hwaccels` を問い合わせ、Sharp/libvipsの `sharp.format` と合わせてキャッシュします。アップロード後はSharp metadata、ffprobe、file-type、実デコード結果を照合し、このサーバーで本当に処理できるファイルだけを受け付けます。

実装済みの主な機能:

- Sharpが実際に読めるJPEG、PNG、WebP、AVIF、GIF、TIFF、SVGなどの画像入力
- FFmpegが実際に読めるMP4、MOV、MKV、WebM、AVI、MPEG、MTS、TS、WMV、FLVなどの動画入力
- FFmpegが実際に読めるMP3、AAC、M4A、WAV、FLAC、Opus、OGG、WMA、AIFFなどの音声入力
- PNG / JPEG / WebP / AVIF / TIFF / GIF、MP4 / WebM / MKV / MOV、MP3 / AAC / Opus / FLAC / WAV出力
- 完全無劣化・高画質最適化・容量優先・アーカイブの4モードと、従来の形式変換・補正・メタデータ処理
- KB・MB・GBまたは元容量比率で指定する目標容量モード、事前推定、画像二分探索、動画2パスエンコード
- Real-ESRGAN画像2倍/4倍、GFPGAN顔補正、フレーム分解型の動画AI超解像のオプション連携
- 画像比較スライダー、等倍・拡大表示、動画の短尺左右プレビュー、各種メディア情報比較
- 進捗段階、進捗率、AI同時処理キュー、キャンセル、解析結果からの確認式おすすめ
- EXIF / GPS / XMP / IPTCおよびコンテナメタデータの削除、期限付きダウンロード

AI依存関係とモデルは自動ダウンロードしません。未導入環境では通常処理を維持し、AI設定だけを理由付きで無効表示します。

## 必要環境

- Node.js 22.22.0（`.nvmrc` / `.node-version`で固定、対応範囲は22系）
- npm 10.9.4（`packageManager`で固定）

動画処理には `ffmpeg-static` が同梱するFFmpegを使うため、通常はシステムへのFFmpegインストールは不要です。デプロイ先ではネイティブ実行ファイルを起動できるNode.jsランタイムが必要です。

任意のAI処理を使う場合だけ、別途Python、PyTorch、Real-ESRGAN、OpenCV、モデル重みが必要です。GFPGAN顔補正にはgfpganとモデル重みも必要です。GPUがある場合はCUDA対応PyTorchを使用します。

## 使い方

1. 画像・動画・音声をドロップします。拡張子ではなく、サーバーが実形式とデコード可否を解析します。
2. 解析後に表示されるおすすめは必要なものだけ「反映する」を押します。自動適用はされません。
3. 完全無劣化、高画質最適化、容量優先、アーカイブから処理モードを選びます。従来の形式変換、画質補正、メタデータ削除も詳細設定から利用できます。
   容量上限が決まっている場合は「目標容量を指定」を選び、プリセット、数値、KB・MB・GB、または元容量比率を設定します。
4. メディア種別ごとの出力、品質、解像度、コーデック、通常補正、任意のAI設定を確認します。処理速度は「高速」「バランス」「最大圧縮」から選択できます。バランスは従来設定を維持します。
5. 「ファイルを最適化する」を押します。動画・音声は進捗と段階を表示し、途中でキャンセルできます。

## 画面構成と操作フロー

操作は「1. ファイルを追加 → 2. 最適化方法を選択 → 3. 処理・結果を確認」の3段階です。

- `/`: ドラッグ＆ドロップ、複数選択、画像のクリップボード貼り付け
- `/optimize`: 解析、おまかせ設定、画質プリセット、必要な場合だけ開く詳細設定
- `/processing/[jobId]`: 実測進捗、工程、推定残り時間、キャンセル
- `/result/[jobId]`: 容量・解像度・形式比較、プレビュー、ダウンロード
- `/history`: この端末に保存した処理履歴。期限切れ項目は自動的に除外
- `/settings`: ライト／ダーク、言語、完成ファイルの保存期間

詳細設定は初期状態で閉じています。画像に動画コーデックを表示しないなど、解析したファイル種別に関係する設定だけを表示します。推定出力サイズは目安であり、実際の結果は入力内容とエンコーダーによって変わります。

## 処理速度とベンチマーク

- 高速: 画像の品質値・動画CRF・品質下限を維持したままエンコーダーの努力量を下げます。短い実エンコードで利用可能と確認できた場合だけハードウェアエンコーダーを使い、失敗時は同じ設定のCPUエンコーダーへ戻ります。目標容量動画は1パスのため、2パスより実容量の誤差が大きくなる場合があります。
- バランス: Sharpの従来努力値、動画の`medium`を維持します。目標容量動画は1パスで処理し、容量精度より処理時間との均衡を優先します。
- 最大圧縮: 画像の高努力値、動画の`slow`相当、目標容量の2パスを使用します。

FFmpegのリサイズ、ノイズ除去、シャープ化、色補正は1つの`-vf`チェーンと1回のFFmpeg実行へ統合しています。通常動画ではPNGフレームを生成せず、AI超解像を選択した場合だけフレーム分解します。

固定ベンチマークは次のコマンドで再現できます。素材は`.benchmark`へ生成され、結果は個人情報・元ファイル名・絶対パスを含まないJSON Linesとして`benchmarks/results`へ保存されます。

```bash
npm run benchmark:generate
npm run benchmark -- --label local-balanced --speed balanced
npm run benchmark -- --label local-fast --speed fast
```

## 進捗表示とリアルタイム通知

処理画面では、ファイルごとの進捗と複数ファイル全体の進捗を分けて表示します。全体カードには完了件数、現在のファイル、処理段階、進捗率、経過時間、推定残り時間、現在までの削減容量を表示します。個別カードではステップ、元サイズ、現在または完成後のサイズ、削減率、動画のフレーム・処理位置・FPS・速度などを確認できます。

進捗はブラウザだけで時間経過に合わせて増やす疑似表示ではありません。画像処理の工程、FFmpegやPythonワーカーから通知された実測値、出力ファイルの検証結果をサーバー側のジョブ状態へ反映します。処理中の値は0〜99%に制限し、出力ファイルが存在して正常に検証できた後だけ100%と完了状態を通知します。

各工程には `pending`、`processing`、`completed`、`failed`、`cancelled` の状態があります。画面はアイコンとテキストを併用し、進捗バーには `role="progressbar"` とARIAの現在値・最小値・最大値、状態通知には `aria-live` を設定します。失敗時には内部パスや生のコマンドを表示せず、利用者向けの説明と再試行手段を表示します。

### Server-Sent Events（SSE）

進捗通知はジョブごとのSSEエンドポイントを優先して使用します。SSEはサーバーからブラウザへの単方向通知に適しており、進捗更新のために短い間隔でAPIへ問い合わせ続ける必要がありません。イベントにはジョブID、ファイルID、状態、工程、進捗率、経過時間、推定残り時間、実測または推定出力サイズなどを含みます。

イベントには単調増加するIDを付けます。通信が切れた場合はブラウザが再接続し、最後に受信したイベントIDまたはジョブIDから状態を復元します。同じイベントを重複受信しても状態が巻き戻らないよう、イベントIDと更新時刻を基準に処理します。SSEを利用できない環境ではジョブ状態APIによる再取得をフォールバックとして使います。ページを再読み込みした場合も、保持中のジョブIDから最新の進捗を取得できます。

開発中にSSEを確認する場合は、処理開始時に返されたジョブIDを使います。

```text
GET /api/jobs/{jobId}/events
Accept: text/event-stream
```

SSEレスポンスは `Cache-Control: no-cache, no-transform` とし、すべての状態更新を `progress` イベントとして次の形式で送信します。完了・失敗・キャンセルはdata内の `status` で判定し、終端イベントを送った後にストリームを閉じます。

```text
id: <単調増加するイベントID>
event: progress
data: {"jobId":"...","status":"encoding","progress":68,...}
```

再接続時はブラウザが送る `Last-Event-ID` を使い、その次の保持イベントから再送します。ページ再読み込み時はジョブIDから保存済み状態とイベント列を読み戻します。イベントIDとsequence以下の更新は無視するため、ネットワーク再送で同じイベントが届いても進捗は巻き戻りません。一定時間進捗がない場合も接続維持用コメントを送りますが、これは進捗率を増加させません。履歴は直近400イベントに制限されます。

ジョブ状態と直近イベントは一時ジョブ内の `job-state.json` にも保存するため、ページ再読み込みや同一サーバー内のメモリ状態消失後も表示を復元できます。ただし、一時保存期限を過ぎた場合、ファイルが清掃済みの場合、またはサーバープロセス自体が処理途中で終了した場合にエンコード処理そのものを途中から再開することはできません。本番で複数Nodeプロセスを動かす場合は、SSEの再接続先にかかわらず同じ状態を参照できるよう、Redisやデータベースへジョブ状態とイベント列を保存してください。

### FFmpegの進捗取得

動画・音声処理では、標準エラー出力の表示文に依存せず、FFmpegへ次の引数を渡して機械可読な進捗を受け取ります。

```text
-progress pipe:1 -nostats
```

`frame`、`fps`、`bitrate`、`total_size`、`out_time_ms` / `out_time`、`speed`、`progress` を読み取り、ffprobeが取得した総再生時間と現在の処理位置から進捗率を計算します。総フレーム数が得られる場合はフレーム情報も詳細表示に使い、得られない場合は再生時間を基準にします。不正値や総時間を超える値は0〜100%へ制限します。

推定残り時間は経過時間と実測進捗から計算し、直近の測定値を平滑化して急激な変動を抑えます。処理開始直後やデータ不足時は「推定時間を取得しています」と表示し、値を表示する場合も「推定」または「約」を付けます。コーデック、入力内容、CPU/GPU負荷によって速度は変わるため、完了時刻を保証する値ではありません。

複数ファイルの全体進捗は個別進捗率の平均とし、完了済み件数は別に集計します。削減済み容量は、出力サイズが確定した完了ファイルだけを対象にして、元サイズ合計から出力サイズ合計を差し引きます。未完了ファイルの推定容量は削減済み容量へ混ぜません。

### キャンセルと復元

キャンセルボタンを押すと確認ダイアログを表示し、確認後にジョブのキャンセルAPIを呼びます。サーバーは対象ジョブのAbortSignalを通知し、実行中のFFmpegまたはPythonワーカーを安全に終了して、状態を `cancelled` に変更します。一時フレーム、抽出音声、中間出力は成功時と同様に `finally` で削除します。キャンセル後に遅れて届いた進捗イベントで処理中へ戻らないよう、`completed`、`failed`、`cancelled` は終端状態として扱います。

通信切断は処理キャンセルとは扱いません。ブラウザはSSEへ再接続し、SSEが利用できない場合は `GET /api/jobs/{jobId}` で最新スナップショットを取得します。明示的なキャンセルは `DELETE /api/jobs/{jobId}` で行い、終了後のイベントを受信した場合も `cancelled` から別状態へ変更しません。

削減率は `(元の容量 - 処理後の容量) / 元の容量 × 100` で計算します。出力が大きくなった場合は負の削減率を隠さず「容量が増加しました」と表示します。0バイト入力は計算せず、アップロード検証でエラーにします。

動画の「無劣化コピー」では、FFmpegに次の指定を渡し、映像・音声を再エンコードせずにメタデータを削除します。

```text
-map 0 -c copy -map_metadata -1 -map_chapters -1
```

動画を追加すると、アップロード直後にffprobeでメディア情報を解析します。「無劣化コピー」では上記のストリームコピーを使い、「動画を圧縮」では解像度・コーデック・品質・音声を指定して再エンコードします。

## 画像の出力形式

設定画面の「出力形式」では、解析済みの実際のメディア種別に応じた形式だけを表示します。画像・動画・音声が混在していてもファイルごとに個別指定でき、同じメディア種別を選択した場合は一括適用もできます。一般ファイルや形式を確認できないファイルには変換候補を表示しません。初期候補は互換性、容量、画質、透過、編集、長期保存などの用途を示しますが、ユーザーがいつでも変更できます。

FFmpegを使う動画・音声形式は、初回利用時に `ffmpeg -muxers`、`ffmpeg -encoders`、`ffmpeg -formats` を取得してキャッシュし、現在の環境で利用できる形式とコーデックだけを選択可能にします。この一覧はジョブごとには取得しません。任意文字列はFFmpeg引数や拡張子に使わず、共通ホワイトリストの形式定義だけを受け付けます。

| 出力形式 | 圧縮方式         | 品質指定          | 透過   | 向いている用途               |
| -------- | ---------------- | ----------------- | ------ | ---------------------------- |
| PNG      | lossless固定     | なし              | 対応   | ロゴ、図版、透過素材         |
| JPEG     | lossy固定        | 1〜100            | 非対応 | 写真、互換性を重視する共有   |
| WebP     | lossless / lossy | lossy時のみ1〜100 | 対応   | Web掲載、軽量化              |
| AVIF     | lossless / lossy | lossy時のみ1〜100 | 対応   | より高い圧縮率を求める場合   |
| TIFF     | LZW lossless     | なし              | 対応   | 編集素材、アーカイブ         |
| GIF      | 256色パレット    | なし              | 対応   | 短いアニメーション、簡易図版 |

PNGはパレット量子化を使わず、画素を保つ可逆圧縮で出力します。JPEG・WebP・AVIFのlossyでは、画面の品質スライダーの値をそのままsharpへ渡します。WebP / AVIFのlosslessでは品質値を使わないため、スライダーは表示されません。

### 変換時の注意点

- 透過を含む画像をJPEGへ変換すると、透明部分は選択した白・黒・カスタム背景色に合成されます。画面と処理結果の両方に警告を表示します。
- JPEGをPNGへ変換しても元の圧縮劣化は改善されず、容量が増える場合があります。
- AVIFは高い圧縮率が期待できますが、特にlosslessでは変換に時間がかかります。
- すべての画像変換で `autoOrient()` を先に適用し、EXIF Orientationを画素へ反映してからEXIF・GPS・XMP・IPTCを削除します。
- 完成ファイルの拡張子とContent-Typeは選択形式に合わせて `.png`、`.jpg`、`.webp`、`.avif`、`.tiff`、`.gif` に変更します。
- GIFは最大256色へ量子化するため、完全無劣化としては扱いません。
- ブラウザが直接表示しにくいTIFF等も、比較用WebPプレビューを別に生成します。

APIには出力形式・圧縮方式・品質値の検証を実装しています。不正な設定、破損画像、未対応形式では、ジョブの一時ファイルを削除して分かりやすいエラーを返します。

従来の「メタデータ削除のみ」も利用できます。この場合は出力形式の設定を使わず、元の形式を維持します。JPEGはEXIF Orientationが未使用なら圧縮済み画像データを再エンコードせず、APP1（EXIF / XMP）、APP13（IPTC）、コメントセグメントを直接除去します。向きの反映が必要な画像では、正しい表示方向を保つために `autoOrient()` 後に出力します。

### 画像の通常補正とAI

通常補正はSharpで、自動回転、sRGB変換、ノイズ軽減、ガンマ、明るさ、コントラスト、彩度、シャープ化の順に適用します。透過画像のノイズ軽減ではalphaを分離して保護します。アニメーションではフレーム保持を優先し、対応できない補正を警告付きで省略します。

Real-ESRGANは写真/アニメモデル、2倍/4倍、圧縮ノイズ除去、弱/標準/強を選べます。弱/標準はLanczos拡大結果とAI結果をブレンドするアプリ独自の強度です。GFPGANは顔補正なし/弱/標準/強を選べます。Pythonワーカーは常駐し、初回処理で読み込んだモデルを同じ設定の後続ジョブで再利用します。CUDAではFP16と`inference_mode`を使い、GPUメモリ量からタイルを選びます。どちらも元データにない細部を推測する場合があるため、証拠・医療・記録用途では処理前後を必ず確認してください。

この環境でPythonモジュールやモデルが見つからない場合、`/api/capabilities` は `available:false` 相当の状態と理由を返し、画面ではAI操作を無効化します。通常のSharp処理には影響しません。

## 動画圧縮

### 出力コンテナとコーデック

| コンテナ  | 拡張子  | 映像コーデック            | 音声コーデック    |
| --------- | ------- | ------------------------- | ----------------- |
| MP4       | `.mp4`  | H.264 / H.265 / AV1       | AAC               |
| WebM      | `.webm` | VP9 / AV1                 | Opus / Vorbis     |
| Matroska  | `.mkv`  | H.264 / H.265 / VP9 / AV1 | AAC / Opus / FLAC |
| QuickTime | `.mov`  | H.264 / H.265             | AAC / PCM         |

コンテナ変更で現在のコーデックが利用できなくなった場合は、利用可能な推奨コーデックへ切り替えて理由を表示します。変換後はffprobeでコンテナとコーデックを再解析し、選択内容と一致しない出力や0バイト・破損出力をダウンロード対象にしません。

### 解像度

次の出力高さを選択できます。幅はFFmpegの `scale=-2:高さ` で自動計算し、アスペクト比と偶数幅を維持します。

- 元のまま
- 2160p / 1440p / 1080p / 720p / 480p
- カスタム高さ（144〜4320pxの偶数）

元動画より大きいプリセットは選択できません。カスタム高さではアップスケールも指定できますが、画質が改善しないことを画面と処理結果に警告します。

### コーデックとCRF

動画圧縮では `preset medium` を使用し、選択した品質を次のCRFへマッピングします。

| コーデック       | 高品質 | バランス | 容量優先 | 特徴                                               |
| ---------------- | -----: | -------: | -------: | -------------------------------------------------- |
| H.264 / libx264  |     18 |       23 |       28 | 互換性を優先。幅広い環境で再生可能                 |
| H.265 / libx265  |     22 |       26 |       30 | 高圧縮。一部のブラウザや端末では再生不可の場合あり |
| VP9 / libvpx-vp9 |     24 |       31 |       38 | WebM向け。圧縮率とWeb互換性のバランス              |
| AV1 / libaom-av1 |     24 |       30 |       36 | 高圧縮だがエンコード負荷が高い                     |

H.265をMP4 / MOVへ出力する場合は `hvc1` タグを付けます。再エンコード時の画素形式は互換性を考慮して `yuv420p` です。

### 音声

- 元のまま: `-c:a copy`
- AAC 128kbps: `-c:a aac -b:a 128k`
- AAC 96kbps: `-c:a aac -b:a 96k`
- Opus 128kbps / 96kbps: WebMなどに適した高圧縮音声

### 無劣化モードとの違い

「無劣化コピー」は解像度を元のままにし、映像・音声を `-c copy` でコピーします。メタデータ削除を有効にした場合は `-map_metadata -1 -map_chapters -1` を追加します。高速で画質を変えませんが、容量はメタデータ分しか減りません。

「動画を圧縮」は解像度が元のままでもCRF指定で再エンコードするため、完全な無劣化ではありません。解像度変更を選んだ場合も必ず再エンコードします。処理中はFFmpegの `-progress pipe:1` とffprobeの再生時間から実際の進捗率を計算します。

変換完了後には、処理前後のファイルサイズ・削減率・解像度・動画コーデック・動画ビットレート・音声コーデック・音声ビットレートを表示します。

### 動画補正、プレビュー、AI超解像

通常補正では `scale`（Lanczos）、`hqdn3d` / `nlmeans`、`unsharp` / `cas`、`eq`、任意の `colorspace` を許可済み引数だけから組み立てます。アスペクト比、元FPS、音声を維持します。処理前後には先頭最大5秒の比較用MP4を生成し、左右で再生できます。

元動画より大きい解像度ではLanczos単純拡大またはReal-ESRGAN AI超解像を選べます。AIではFFmpegでPNGフレームへ分解し、1回起動したPythonワーカーで順番に高画質化し、元FPSで再構築した後、抽出しておいた元音声を再結合します。音声なし動画にも対応し、成功・失敗・キャンセルのすべてで一時フレームを `finally` 削除します。

AI動画は入力100MB以下、Full HD以下、GPU時60秒/CPU時10秒に制限しています。AI同時処理数は既定1で、`AI_MAX_CONCURRENCY` は安全上2までです。進捗段階と状態はメモリおよびジョブディレクトリの `job-state.json` に保存し、`DELETE /api/jobs/:jobId` でキャンセルできます。単一Nodeプロセス向けの実装なので、複数インスタンス運用ではBullMQ/Redisなどの共有キューに置き換えてください。

## 高度な最適化モード

### 完全無劣化

「無劣化」は検証に成功した候補だけに表示します。単にlosslessエンコーダーを指定しただけでは無劣化扱いにしません。

- PNG: OxiPNG、ZopfliPNG、Sharp PNG、WebP losslessの利用可能な候補を生成します。ICC、ガンマ、色空間など表示に関係する安全なチャンクを保持し、sRGB RGBAへデコードしたSHA-256が一致した候補だけを採用します。
- JPEG: Sharpで再エンコードせず、jpegtranのoptimize・progressive・メタデータ保持・プライバシー情報削除候補を比較します。デコード後RGBAが一致した候補だけを採用します。
- JPEG XL: ユーザーが明示的に有効化し、cjxlとdjxlが利用できる場合だけJPEG可逆トランスコード候補を作ります。djxlで復元したJPEGのバイトSHA-256一致を確認します。JXLはブラウザや編集ソフトの互換性に注意が必要です。
- 動画: FFmpegの `-c copy` で保持対象の圧縮済み映像・音声ストリームをコピーし、入力と出力のstreamhash SHA-256を比較します。副音声、字幕、添付、チャプター、プライバシーメタデータはユーザーが事前一覧を確認してから削除できます。HDR、色空間、回転、アスペクト比、FPSに必要なストリーム情報は一律削除しません。

OxiPNG、ZopfliPNG、jpegtran、cjxl、djxlは任意導入です。起動時の固定対応表ではなく、実行ファイルを安全な引数で起動できるか動的に確認し、未導入の方式は結果の候補一覧に「利用不可」と理由を表示します。任意ツールがなくても元ファイル保持とSharpで検証可能なPNG/WebP候補は利用できます。

### 高画質最適化

動画はAV1の複数CRF、H.265の複数CRF、H.264の高品質CRF候補を順番に生成します。元解像度・元FPS・音声stream copyを維持し、10bit/HDR入力では画素形式、色域、伝達特性、回転、SARを候補へ引き継ぎます。`medium`、`slow`、`slower` presetを選択できます。

FFmpegにlibvmafがある場合、各候補について平均VMAF、最低フレームVMAF、しきい値を下回った連続区間を取得します。平均と最低値の両方が指定基準を満たす候補だけを「高画質基準を満たした候補」とし、その中で最小のものを採用します。VMAFは完全な画質保証ではないため、結果画面の比較プレビューでも確認してください。libvmafがない環境では虚偽の高画質判定を行わず、候補を利用不可として元ファイルを採用します。

### 容量優先とアーカイブ

- 容量優先: 画質変化を許容する既存の圧縮設定に加え、動画では選択したVMAF基準を満たす候補を探索します。
- アーカイブ: 画像は可逆候補、動画はストリームコピーを優先し、長期保存で元データを維持しやすい方式を使います。JPEG XLは明示選択時だけ候補に含めます。

すべてのモードで元ファイルは候補として保持します。検証済み出力が元より大きい場合は元ファイルを採用し、その理由を結果に表示します。ユーザーが字幕やプライバシー情報の削除を明示し、削除済み候補が大きくなる場合は、削除指定を優先したことを結果へ明記します。結果レポートには元・出力容量、削減率、採用方式、形式・コーデック、無劣化検証、VMAF、不採用候補、元ファイル維持理由を表示します。

## 目標容量モード

完全無劣化とは別のモードです。目標容量へ近づけるため再エンコードする場合がありますが、設定した品質下限を超えて画質や音質を下げません。メール添付、SNS共有、Web掲載、スマートフォン、クラウド、元容量の半分、100MB以下、カスタムのプリセットは `lib/target-size/config.ts` に集約しています。コンポーネント内に容量値を固定していません。上記の環境変数または設定ファイルで変更できます。

数値はKB、MB、GBで指定でき、比率プリセットはファイルごとに元容量から目標バイト数を求めます。目標が元容量以上の場合は再エンコードせず、圧縮不要と表示して元ファイルを返します。

### 動画の計算と2パスエンコード

ffprobeで再生時間、元容量、映像・音声ビットレート、解像度、FPS、映像・音声コーデック、音声トラック数を取得します。計算は次の順です。

```text
総ビットレート = 目標容量(byte) × 8 ÷ 再生時間(秒)
使用可能ビットレート = 総ビットレート − 安全マージン − コンテナオーバーヘッド
映像ビットレート = 使用可能ビットレート − 音声ビットレート × 音声トラック数
```

安全マージンは既定3%、コンテナオーバーヘッドは既定1.5%です。音声は元ビットレートを超えない範囲で320、256、192、128、96、64kbpsの順に選び、映像品質を確保できない場合だけ段階的に下げます。動画では音声削除も選択できます。

H.264、H.265、利用可能な場合はAV1を、高速・バランスでは1パス、容量精度を優先する最大圧縮では2パスで処理します。2パスでは1回目で映像を解析し、2回目で計算済みビットレートと音声を出力します。どの速度でも品質下限は変更しません。Windowsは `NUL`、macOS/Linuxは `/dev/null` を1パス目のnull出力に使います。passlogは検証済みジョブIDごとに分け、成功・失敗・キャンセルのすべてで `finally` から削除します。FFmpegは `shell: false` と列挙済み引数配列だけで起動します。

元解像度で最低映像ビットレートを確保できない場合は、元より高くない2160p、1440p、1080p、720p、480pの順から候補を作ります。初期状態では提案だけを表示し、「解像度変更を許可」をユーザーが選ぶまで自動縮小しません。

長い動画では「先頭・中間・終盤で精度を上げる」を押すと、各区間の短いサンプルを実エンコードし、全体容量と処理時間を「推定」「約」として更新できます。詳細推定は任意で、高速・バランスの本処理を待たせません。最大圧縮を選んだ場合だけ高精度な2パス処理を使用します。

### 画像の二分探索

JPEG、WebP、AVIFは、品質下限を先に生成して目標達成可能性を確認してから、下限と最高品質の中央を試す二分探索を行います。目標以下になった最高品質候補を採用し、既定8回または目標との差2%以内で停止します。すべての候補はSharpで再デコードできることを確認し、元より大きい候補は採用しません。

PNGはPNG可逆最適化、WebP lossless、AVIF losslessの順で比較し、ユーザーが「PNGの非可逆圧縮を許可」を選んだ場合だけ非可逆探索へ進みます。透過画像をJPEGへ変換する場合は背景色指定が必須です。

### 品質下限と未達時の扱い

既定の下限はJPEG 60、WebP 55、AVIF 45、動画480p、音声64kbpsです。目標達成にこれらを下回る必要がある場合は無理に処理を続けません。結果には指定容量、実出力、差、達成成否、探索回数、採用品質・解像度・コーデック・音声、現在の下限での推定最小容量と次の選択肢を表示します。

- 高画質を維持できる最小容量を使う
- 品質下限を下げる
- 解像度を下げる
- 音声品質を下げる、または動画音声を削除する
- 目標容量を変更する

目標を超えた出力は成功扱いにせず、元ファイルまたは検証済みの最良候補を返して理由を表示します。出力動画・音声はffprobe、画像はSharpの再デコードで破損していないことを検証します。

## 音声変換

音声専用ストリームはffprobeでコーデック、時間、ビットレート、サンプルレート、チャンネル数を取得します。実行中のFFmpegにエンコーダーとmuxerがある場合だけ、MP3、M4A、AAC、Opus、Ogg、WAV、FLACを表示します。MP3/M4A/AAC/Opus/Oggは高品質・バランス・容量優先のビットレートへ対応し、FLAC/WAVは無劣化出力です。

| 形式 | 拡張子  | エンコーダー | MIMEタイプ   |
| ---- | ------- | ------------ | ------------ |
| MP3  | `.mp3`  | libmp3lame   | `audio/mpeg` |
| M4A  | `.m4a`  | AAC          | `audio/mp4`  |
| AAC  | `.aac`  | AAC          | `audio/aac`  |
| Opus | `.opus` | libopus      | `audio/opus` |
| Ogg  | `.ogg`  | libvorbis    | `audio/ogg`  |
| WAV  | `.wav`  | PCM          | `audio/wav`  |
| FLAC | `.flac` | FLAC         | `audio/flac` |

ダウンロード名は内部の一時保存名と分離し、常に `元のベース名_comp.選択した拡張子` とします。APIは選択形式のContent-Typeと、日本語対応のContent-Dispositionを返します。結果画面とダウンロードボタン付近にも実際の保存予定名を表示します。

メタデータだけ削除する場合は、対応する元コンテナへ `-c:a copy -map_metadata -1 -map_chapters -1` できるときだけ実行します。WMAなど対応出力へストリームコピーできない入力では、形式変換を選ぶよう明確なエラーを返します。

## ファイルサイズ制限

- 画像: 1ファイル 25MBまで
- 音声: 1ファイル 100MBまで
- 動画: 1ファイル 250MBまで
- 1回の選択: 10ファイルまで
- 通常メディア: 30分まで、動画7680×4320まで
- 画像: 4000万入力画素、120フレーム、総8000万フレーム画素まで

ブラウザ側とAPI側の両方で検証し、APIでは拡張子ではなくファイル内容のシグネチャも確認します。リバースプロキシやホスティング環境を使う場合は、その環境のリクエストサイズ上限とタイムアウトも合わせて設定してください。

## 技術構成

- フロントエンド: Next.js App Router、React、TypeScript、Tailwind CSS
- 画像処理: sharp
- AI画像/動画: Python、PyTorch、Real-ESRGAN、GFPGAN（任意導入）
- メタデータ解析: exifr
- 動画処理: FFmpeg（ffmpeg-static）
- 動画情報解析: ffprobe（ffprobe-static）
- 音声処理: FFmpeg

主要な責務は次のように分離しています。

```text
app/
  api/capabilities/     実行環境の動的能力一覧
```

`lib/storage` をストレージ境界としているため、将来はS3アダプターへ差し替えやすい構成です。処理結果にジョブIDを持たせているため、データベースを追加すれば処理履歴やユーザー単位の所有権も実装できます。認証はAPIの前段にセッション検証を加える想定です。

## 一時ファイルの扱い

- 画像のアップロード元は変換処理が終わるとすぐ削除します。
- 動画はffprobe解析後にジョブID付きで一時保存し、変換完了時または失敗時に元ファイルを削除します。
- 完成ファイルと最小限のマニフェストだけを30分保持します。
- プロセス内タイマーに加え、APIアクセス時にも期限切れジョブを清掃します。
- 処理失敗時はジョブディレクトリ全体を削除します。
- AI動画のフレーム、抽出音声、中間動画は成功・失敗・キャンセルのすべてで削除します。

複数インスタンスで運用する場合、ローカル `/tmp` は共有されません。その段階でS3などの共有ストレージと、キューまたは定期削除ジョブへ置き換えてください。

## セキュリティ

- PE / ELF / Mach-O / shebangのmagicを検出して実行ファイルを拒否
- 元ファイル名を表示用だけに使い、保存名はUUIDジョブと `source.bin` に固定
- UUIDと`basename`を再検証し、パストラバーサルを防止
- SVGのDOCTYPE、ENTITY、script、イベント属性、外部URL参照を拒否
- Sharpの画素・寸法・フレーム上限、ffprobe/FFmpeg/Pythonの時間・出力上限を設定
- `spawn(..., { shell: false })` を使用し、列挙検証済み値だけから引数配列を生成
- リモートプロトコルを許可せず、ローカルファイルだけを解析
- `finally`とジョブディレクトリ削除でエラー時にも一時データを清掃

## ダウンロードファイル名

処理結果の初期保存名は、画像・動画・音声で共通して`元のベース名_comp.実際の出力拡張子`です。たとえば`photo.jpg`をWebPへ変換すると`photo_comp.webp`、`資料.最終版.png`をAVIFへ変換すると`資料.最終版_comp.avif`になります。最後の拡張子だけを外すため、複数のドット、日本語、スペース、ハイフン、アンダースコアは維持されます。

共通処理は`shared/files/create-compressed-file-name.ts`に集約しています。`< > : " / \\ | ? *`、制御文字はアンダースコアへ置換し、末尾のピリオドと空白を除去します。空のベース名は`compressed_file`になります。複数ファイル用の名前は同じ関数を使い、衝突時に`_2`、`_3`を追加します。ZIP本体名は`compression_files_comp.zip`です。

サーバー内部では元ファイル名をパスへ使わず、UUIDのジョブディレクトリ、`source.bin`、`output.実拡張子`で管理します。表示用の元名、ユーザー向け保存名、内部ファイル名はマニフェスト上でも分離しています。ダウンロードAPIは`Content-Disposition`へASCII用`filename`と日本語対応の`filename*=UTF-8''...`を設定し、改行や危険文字を除去します。画面のダウンロードリンクは同じ保存名を`download`属性へ設定します。Blobを使う共通ダウンロード処理はクリック後に`URL.revokeObjectURL`でObject URLを必ず解放します。

ウイルススキャナーそのものは同梱していません。公開サービスではClamAV等、認証、IP/ユーザー単位レート制限、CPU/GPUクォータも併用してください。

## 無劣化と「視覚劣化なし」の違い

- 無劣化: 画像はlosslessエンコーダーだけ、動画・音声は可能なら `-c copy` を使い、再エンコードが必要な補正・リサイズ・非可逆形式をUIとAPIで拒否します。
- 視覚劣化なし相当: 高品質なJPEG/WebP/AVIFや低CRF動画で、人が通常の表示条件では差を認識しにくい設定です。数学的には元データと同一ではありません。
- 画質改善: ノイズ軽減・シャープ化・色補正・AI推定により見た目を変えます。改善に見えても元データの忠実な保存ではありません。

## 本番運用へ向けた拡張候補

- S3互換ストレージとライフサイクル削除
- BullMQ / Redisによる複数インスタンス対応キューとGPUワーカープール
- 処理履歴、再ダウンロード、ユーザー認証

## リファクタリング時の設計方針

- UIは表示と操作に集中させ、容量計算、候補探索、外部コマンド引数生成、出力検証は純粋関数または専用サービスへ置きます。
- 依存方向は `app / components → features → shared` を基本とし、FFmpeg、ffprobe、Sharp、ファイルシステムなどの実装詳細は `infrastructure` へ閉じ込めます。
- `lib/errors.ts` は既存コード向けの互換入口です。新しい処理では具体的なエラー分類を使い、利用者向け文言と内部診断を分離します。
- ログは `shared/logging` を経由し、許可する属性を `jobId`、`fileId`、`stage`、`errorCode`、`elapsedMs` に限定します。元ファイル名、絶対パス、外部ツールの生ログ、位置情報は記録しません。
- 外部ツール境界は実行関数を注入可能にし、回帰テストでは実バイナリを起動せずに成功、失敗、タイムアウト、キャンセルを検証します。
- CIでは lint、型検査、テスト、ビルドを順に実行し、段階的な移行中も既存の入出力契約を保護します。

## 残っている技術的負債

- 一部の大きな画面・APIオーケストレーターは段階移行中です。機能単位のuse caseへ分け、React／Next.js依存のない処理をさらに増やす余地があります。
- `lib/errors.ts` の汎用 `AppError` は互換性のため残しています。既存の文字列コードは、変更対象に触れる際に具体エラーへ置き換えます。
- ジョブ状態とAIキューは単一Nodeプロセスを前提とします。複数インスタンス運用では共有キューと共有状態ストアが必要です。
- ログの出力先は現在標準出力です。本番では同じ属性許可リストを維持する構造化ログ基盤へsinkを差し替えてください。
- 外部ツールの詳細診断は利用者レスポンスと標準ログから隔離しています。アクセス制御と保存期限を備えた診断ストアは未実装です。
